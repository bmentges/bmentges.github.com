            <section class="content-section">
                <p>Over the past two years, I integrated LLM-powered agents into the data engineering workflows of a major B2B data platform — not as a side tool, but as a core part of how we built, maintained, and operated pipelines processing terabytes daily.</p>
                <p>I built five agents: pipeline scaffolding, code review, incident response, documentation generation, and code generation. I wired them together with MCP (Model Context Protocol) so they could access our live infrastructure — schemas, DAG definitions, run histories, team conventions.</p>
                <p>They were useful. Some were genuinely transformative. But every single one hit the same ceiling, and it took me months to understand what it was.</p>
                <p>This is the honest account — what worked, where the ceiling was, and the thing I'd now do <em>first</em> on any new engagement.</p>
            </section>

            
            <section class="content-section">
                <h2>The foundation: MCP changed how agents see infrastructure</h2>
                <p>The single biggest lever in the first phase was MCP. Before it, using an LLM for our specific codebase meant either dumping thousands of lines into a prompt or accepting generic advice. MCP let me build structured context servers that gave agents exactly what they needed:</p>
                <ul>
                    <li><strong>Pipeline metadata</strong> — DAG definitions, schedule configurations, dependency graphs, recent run histories</li>
                    <li><strong>Schema information</strong> — warehouse table schemas, column types, freshness metadata, lineage</li>
                    <li><strong>Model definitions</strong> — transformation SQL, documentation, test definitions, upstream/downstream dependencies</li>
                    <li><strong>Team conventions</strong> — naming patterns, partition strategies, testing requirements, PR checklists</li>
                </ul>
                <p>With this context, agents went from "here's some generic Python" to "the upstream model refreshes at 06:00 UTC, so your dependency should wait until 06:30 to account for the 15-minute SLA buffer we use." Specificity changed everything.</p>

                <p>Each agent produced real results:</p>
                <ul>
                    <li><strong>Pipeline scaffolder</strong> — cut new pipeline setup from 45-60 minutes to 15 minutes. Consistent naming, proper partitioning, tests pre-populated.</li>
                    <li><strong>Code review assistant</strong> — caught mechanical issues (missing null checks, partition skew risks, schema drift) before human review, freeing senior engineers for architectural feedback.</li>
                    <li><strong>Incident response</strong> — assembled context briefings in seconds instead of 15-20 minutes of manual reconstruction at 3 AM. The most impactful agent for on-call quality of life.</li>
                    <li><strong>Documentation generator</strong> — kept the "what" in sync with code changes automatically. Humans wrote the "why."</li>
                </ul>
                <p>MCP made all of this possible. Without it, agents gave generic advice. With it, they gave answers grounded in <em>our</em> specific infrastructure.</p>
                <p>But then every agent hit the same wall.</p>
            </section>

            
            <section class="content-section">
                <h2>The ceiling: agents could see the system, but they couldn't remember the organization</h2>
                <p>MCP gave agents access to <em>what</em> our system looked like right now — live schemas, current DAG definitions, recent run histories. That's infrastructure data. It's real-time and it's precise.</p>
                <p>But there's an entire category of knowledge that doesn't live in infrastructure:</p>
                <ul>
                    <li><em>Why</em> was this pipeline designed this way? What constraints shaped the decision?</li>
                    <li>The last three times we tried to migrate this service, what went wrong?</li>
                    <li>This failure pattern recurs every quarter-end due to seasonal load spikes — but that's nowhere in the DAG definition</li>
                    <li>The team decided six months ago to standardize on a specific approach — but that's in a Slack thread, not in code</li>
                    <li>The senior engineer who built this system is on leave and no one else fully understands the token rotation mechanism</li>
                </ul>
                <p>This is <strong>institutional knowledge</strong> — the accumulated wisdom of the team, scattered across people's heads, old Slack threads, postmortems, and design docs that nobody maintains. MCP can't see any of it. And every agent I built was limited by its absence.</p>
                <p>The incident response agent could pull run histories and upstream status — but it couldn't know that this exact failure pattern happened last quarter and the root cause was an upstream schema change, not the pipeline itself. The code review agent caught missing null checks — but it couldn't flag that the proposed approach contradicted an architecture decision the team made six months ago. The scaffolder generated correct pipelines — but it couldn't account for a deprecation that existed only in a team planning document.</p>
                <p>Every agent was technically correct and architecturally naive. They could see the system. They couldn't <em>remember</em> the organization.</p>

            <section class="content-section">
                <h2>The breakthrough: a knowledge base as the agent's memory</h2>
                <p>After leaving that engagement, I built the thing I wished I'd had from the start: a structured, LLM-maintained knowledge base that captures the institutional knowledge MCP can't see.</p>
                <p>The concept builds on <a href="https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f" target="_blank" rel="noopener">Andrej Karpathy's LLM Wiki pattern</a>: instead of having agents re-derive understanding every time (the RAG approach), the LLM <strong>incrementally builds and maintains a persistent wiki</strong>. You feed it source documents — architecture decision records, postmortems, design docs, meeting notes — and it creates cross-referenced wiki pages with confidence scoring, contradiction detection, and lifecycle management.</p>
                <p>The knowledge compounds over time. Every source ingested makes every future agent interaction smarter, because the synthesis is already done and maintained.</p>
                <p>I've open-sourced this as a reusable template: <a href="https://github.com/bmentges/knowledge-base-template" target="_blank" rel="noopener"><strong>knowledge-base-template</strong></a>.</p>
                <p>When you combine MCP (live infrastructure) with the knowledge base (institutional memory), agents get the full context picture:</p>

                <p>This isn't theoretical. Here's what each agent looks like with the knowledge base:</p>
                <ul>
                    <li><strong>Incident response</strong> — doesn't just show run histories. It knows this pipeline has a seasonal quarter-end load spike, that the last time it failed this way the root cause was an upstream schema change, and that the team's preferred remediation is to increase partition count rather than cluster size.</li>
                    <li><strong>Code review</strong> — doesn't just catch null checks. It flags that the proposed approach contradicts ADR-047 (the team's decision to avoid cross-service joins), cites the architecture decision, and suggests an alternative that aligns.</li>
                    <li><strong>Scaffolder</strong> — doesn't just follow naming conventions. It knows that Service X is being deprecated next quarter, so it routes the new pipeline through Service Y instead. That deprecation plan is in the knowledge base, not in any config file.</li>
                    <li><strong>Documentation</strong> — doesn't just describe what a pipeline does. It links to the business decision that created it, the incident that shaped its error handling, and the design doc that explains its unusual partition strategy.</li>
                </ul>
                <p>Same agents. Same MCP. But with the knowledge base, they produce work that reflects a year of institutional context instead of a cold read of the codebase.</p>

            <section class="content-section">
                <h2>If I were starting today: knowledge base first</h2>
                <p>This is the key insight, and it inverts the order I followed: <strong>start with the knowledge base, then build agents on top of it.</strong></p>
                <p>Here's the sequence I'd follow on any new engagement:</p>
                <ol>
                    <li><strong>Week 1-2: Build the knowledge base.</strong> Ingest the documents that hold institutional knowledge — ADRs, postmortems, design docs, onboarding guides. Use <a href="https://github.com/bmentges/knowledge-base-template" target="_blank" rel="noopener">the template</a> to set up the three-layer architecture (raw sources, LLM-maintained wiki, schema). Three good ADRs in the wiki will improve agent output more than fifty raw code files.</li>
                    <li><strong>Week 2-3: Set up MCP.</strong> Connect agents to live infrastructure — schemas, orchestrator metadata, run histories. This is the "eyes" layer.</li>
                    <li><strong>Week 3-4: Build the first agent (incident response).</strong> It has the highest impact-to-effort ratio and the fastest team buy-in. With the knowledge base already in place, it's immediately useful — not just assembling context, but <em>understanding</em> it.</li>
                    <li><strong>Week 4+: Add agents incrementally.</strong> Code review, scaffolding, documentation. Each one benefits from the knowledge base that's been compounding since week 1.</li>
                </ol>

            <section class="content-section">
                <h2>The bottom line</h2>
                <p>AI agents in data engineering are production-ready. But their value is determined entirely by the context infrastructure underneath them.</p>
                <p>MCP gives agents eyes into your live infrastructure — the <em>what</em>. A knowledge base gives them memory of your organization — the <em>why</em>. Together, they're the full context layer that turns generic AI into your team's AI.</p>
                <p>If I were starting a new AI integration engagement today, I'd start with the knowledge base. Not the agents. Not the MCP servers. The knowledge base — because it compounds from day one, and every agent you build on top of it is immediately better for it.</p>
                <p>The teams that will win with AI agents aren't the ones with the best models. They're the ones that build the memory infrastructure to make those models genuinely understand their system.</p>
                <p>I've written about the knowledge base strategy in depth: <a href="/posts/knowledge-base-as-agentic-memory"><strong>The Knowledge Base Strategy: Giving AI Agents a Memory That Compounds</strong></a>. And I've open-sourced the template: <a href="https://github.com/bmentges/knowledge-base-template" target="_blank" rel="noopener"><strong>knowledge-base-template</strong></a>.</p>
