            <section class="content-section">
                <p>Most engineering teams using AI today are stuck in the same place: they have a coding assistant, it autocompletes reasonably well, and nobody would call it transformative. The gap between "we have copilot" and "AI is part of our engineering process" is enormous — and it's not a model gap. It's a context gap.</p>
                <p>Agents without context about <em>your</em> system give generic advice. They suggest patterns that contradict your architecture decisions. They scaffold pipelines that ignore your naming conventions. They generate code that doesn't account for the constraints your team learned the hard way over two years of production incidents.</p>
                <p>I've spent the past two years integrating AI agents into data engineering workflows at a major B2B data platform — pipelines processing terabytes daily, dozens of engineers, hundreds of models. The agents that actually shipped and earned team trust weren't the smartest ones. They were the ones with the best context infrastructure underneath them.</p>
                <p>That infrastructure has three layers: <strong>MCP</strong> for live infrastructure access, <strong>guardrails</strong> for trust boundaries, and a <strong>knowledge base</strong> for institutional memory. This article is a field guide to building all three.</p>
            </section>

            
            <section class="content-section">
                <h2>MCP: what it actually is</h2>
                <p>Model Context Protocol is one of those things that sounds abstract until you use it, and then you wonder how you ever worked without it. Let me cut through the marketing.</p>
                <p>MCP is not a framework. It's not a library. It's a <strong>protocol</strong> — a standardized way for LLM agents to access external data sources. Think of it as an API specifically designed for agent context. The agent is the client. Your infrastructure is the server. MCP defines how they talk to each other.</p>
                <p>Before MCP, giving an agent context about your system meant one of two things: dump thousands of lines of code into a prompt (expensive, noisy, hits token limits) or accept that the agent would give generic advice based on its training data. Neither option works for production engineering.</p>
                <p>MCP changes this by letting you build <strong>context servers</strong> — lightweight services that expose specific slices of your infrastructure in a structured format that agents can query on demand. The agent asks "what does the schema of this table look like?" and the MCP server returns the answer. The agent asks "what are the upstream dependencies of this pipeline?" and gets a precise graph. No prompt stuffing. No guessing.</p>
                <p>The client-server model is the key insight. The agent doesn't need to know <em>how</em> your infrastructure works. It just needs to know how to ask questions through the protocol. This means you can swap out infrastructure without changing the agent, and you can upgrade agents without changing your servers.</p>

                <h2>Setting up context servers in practice</h2>
                <p>For a data engineering team, I've found four context servers cover roughly 90% of what agents need. Here's what each one looks like and what makes the difference between a useful server and a noisy one.</p>

                <h3>1. Pipeline/orchestrator metadata server</h3>
                <p>This server exposes your workflow orchestrator's metadata: DAG definitions, schedule configurations, dependency graphs, recent run histories, task durations, and failure rates. When an agent needs to understand how a pipeline is structured or why a run failed, this is where it looks.</p>
                <p><strong>What makes it good</strong>: Expose the <em>operational</em> view, not just the config. An agent doesn't just need to know that Pipeline X runs at 06:00 UTC — it needs to know that Pipeline X's average runtime is 23 minutes, it failed 3 times last month, and its downstream consumers start at 07:00. The difference between config and operations is the difference between a map and ground truth.</p>

                <h3>2. Schema/warehouse metadata server</h3>
                <p>This server exposes your data warehouse structure: table schemas, column types, partition strategies, freshness metadata, row counts, and lineage. When an agent scaffolds a new model or reviews a query, this is its reference.</p>
                <p><strong>What makes it good</strong>: Include freshness and lineage, not just structure. Knowing that a table has 50 columns is useful. Knowing that it was last refreshed 3 hours ago, updates daily at 05:00, and feeds into 12 downstream tables is what changes an agent's output from "technically valid SQL" to "production-aware SQL."</p>

                <h3>3. Code repository context server</h3>
                <p>This server exposes your codebase structure: directory layout, file types, recent changes (last N commits per path), active branches, and code conventions. Not the full code — that's what the agent reads directly. This server provides the <em>metadata about</em> the code.</p>
                <p><strong>What makes it good</strong>: Surface change velocity and ownership. A file that hasn't been touched in 18 months needs different handling than one with 40 commits this quarter. Knowing who last modified a module helps agents understand context even when the code comments are sparse.</p>

                <h3>4. Documentation/conventions server</h3>
                <p>This server exposes your team's standards: naming conventions, testing requirements, PR checklists, style guides, and onboarding documentation. This is the "how we do things here" layer.</p>
                <p><strong>What makes it good</strong>: Be prescriptive, not descriptive. "We use snake_case for table names" is more useful to an agent than "table names vary across the codebase." If the conventions are aspirational rather than actual, say so — agents that enforce conventions the team doesn't actually follow lose trust fast.</p>

                <p>A common mistake: building context servers that return too much data. An agent doesn't need a full table dump — it needs the schema, the freshness timestamp, and the top consumers. <strong>Think about what a senior engineer would look up</strong> when answering a question, and expose that. If a server response requires scrolling, it's too verbose.</p>

            <section class="content-section">
                <h2>The guardrails framework</h2>
                <p>Here's a pattern I've seen kill AI adoption at three different teams: someone builds a capable agent, it does something unexpected in production, and the entire initiative gets shut down. Not because the agent was bad — because there were no boundaries. One misstep costs months of trust.</p>
                <p>Guardrails come <strong>before</strong> capabilities. Before you give an agent the ability to do anything useful, you define what it absolutely cannot do. This sounds backward — why limit the thing you're trying to make powerful? Because trust is the bottleneck, not capability. A limited agent that the team trusts will be adopted. A capable agent that the team fears will be shelved.</p>
                <p>I use a three-zone framework:</p>

                <h3>Zone 1: CAN DO (autonomous)</h3>
                <p>Actions the agent can take without any human approval. These are read-only or low-blast-radius operations:</p>
                <ul>
                    <li><strong>Read data</strong> — query schemas, inspect pipeline metadata, browse run histories</li>
                    <li><strong>Generate code</strong> — produce code snippets, scaffold new files, write tests</li>
                    <li><strong>Suggest changes</strong> — propose PR descriptions, recommend fixes, draft documentation</li>
                    <li><strong>Review PRs</strong> — comment on pull requests, flag issues, check conventions</li>
                </ul>

                <h3>Zone 2: WITH REVIEW (supervised)</h3>
                <p>Actions the agent can prepare but a human must approve before execution:</p>
                <ul>
                    <li><strong>Deploy suggestions</strong> — the agent can draft a deployment plan, but a human runs it</li>
                    <li><strong>Modify configurations</strong> — the agent can propose config changes, but they go through normal review</li>
                    <li><strong>Create tickets/issues</strong> — the agent can draft incident reports or feature requests for human review</li>
                </ul>

                <h3>Zone 3: CANNOT (hard boundary)</h3>
                <p>Actions that are off-limits regardless of context. These are non-negotiable:</p>
                <ul>
                    <li><strong>Modify production data</strong> — no writes to production tables, no deleting records, no schema migrations</li>
                    <li><strong>Merge code</strong> — agents can suggest, review, and approve, but the merge button is human-only</li>
                    <li><strong>Skip tests</strong> — no bypassing CI/CD gates, no overriding quality checks, no "just this once" exceptions</li>
                </ul>

            <section class="content-section">
                <h2>Workflow integration patterns</h2>
                <p>Context servers give agents eyes. Guardrails give them boundaries. The next question is: where do agents actually plug into the engineering lifecycle? Not "where could they theoretically help" — where do they produce measurable value with acceptable risk?</p>
                <p>After building five agents and watching which ones earned adoption and which ones got ignored, I've identified five integration points that consistently work:</p>

                <h3>1. Pipeline scaffolding</h3>
                <p><strong>Trigger</strong>: Engineer starts a new pipeline or model. <strong>Context needed</strong>: Schema server (upstream tables), code server (naming conventions, directory structure), docs server (standards). <strong>Output</strong>: Pre-populated pipeline with correct naming, partitioning, tests, and documentation stubs. <strong>Guardrails</strong>: CAN DO — generates files only, no execution.</p>
                <p>This cut new pipeline setup from 45-60 minutes to 15 minutes. The real win isn't speed — it's consistency. Every scaffolded pipeline follows the team's conventions because the agent reads them from the context server, not from memory.</p>

                <h3>2. Pre-review checks</h3>
                <p><strong>Trigger</strong>: PR opened or updated. <strong>Context needed</strong>: Schema server (affected tables), pipeline server (downstream dependencies), docs server (conventions). <strong>Output</strong>: Comments on the PR flagging mechanical issues — missing null checks, partition skew risks, schema drift, convention violations. <strong>Guardrails</strong>: CAN DO — comments only, cannot approve or merge.</p>
                <p>This freed senior engineers to focus on architectural review instead of catching mechanical issues. The agent handles the "did you remember to..." checklist so humans can focus on "should we..."</p>

                <h3>3. Incident context assembly</h3>
                <p><strong>Trigger</strong>: Alert fires or on-call engineer escalates. <strong>Context needed</strong>: Pipeline server (run history, upstream status, failure patterns), schema server (recent changes to affected tables). <strong>Output</strong>: A context briefing — what failed, what changed recently upstream, similar past failures, affected downstream consumers. <strong>Guardrails</strong>: CAN DO — read-only assembly, no remediation actions.</p>
                <p>This was the most impactful agent. At 3 AM, nobody wants to spend 20 minutes manually checking upstream statuses and cross-referencing recent deploys. The agent assembles the same context in seconds.</p>

                <h3>4. Documentation sync</h3>
                <p><strong>Trigger</strong>: Code merged to main branch. <strong>Context needed</strong>: Code server (diff analysis), schema server (affected models), docs server (existing documentation). <strong>Output</strong>: Updated documentation reflecting code changes — model descriptions, column definitions, dependency notes. <strong>Guardrails</strong>: WITH REVIEW — docs are generated as a PR for human approval before merge.</p>
                <p>Documentation that stays in sync with code is one of those things every team wants and nobody maintains. The agent handles the "what" (this model now has a new column, this pipeline added a dependency) and humans review to add the "why."</p>

                <h3>5. Architecture planning</h3>
                <p><strong>Trigger</strong>: Engineer or tech lead starts planning a new feature or migration. <strong>Context needed</strong>: All four servers plus the knowledge base (architecture decisions, past failures, in-flight projects). <strong>Output</strong>: A structured plan draft with dependency analysis, risk flags, and references to relevant prior decisions. <strong>Guardrails</strong>: WITH REVIEW — the plan is a starting point for human refinement, never executed directly.</p>
                <p>This is where the knowledge base matters most. An agent with only MCP can tell you which tables are involved. An agent with MCP plus the knowledge base can tell you that this migration pattern failed last year and what the team learned from it.</p>

            <section class="content-section">
                <h2>The missing layer: institutional memory</h2>
                <p>MCP gives agents real-time infrastructure data — the <em>what</em> of your system right now. Schema server knows your table structure. Pipeline server knows your DAG definitions. Code server knows your repo layout.</p>
                <p>But there's an entire category of knowledge that doesn't live in infrastructure:</p>
                <ul>
                    <li><em>Why</em> was this pipeline designed this way? What constraints shaped the decision?</li>
                    <li>The last three times the team tried to optimize this query, what approaches failed?</li>
                    <li>This failure pattern recurs every quarter-end due to seasonal data volume spikes — but that's nowhere in the DAG definition</li>
                    <li>The team standardized on a specific partitioning strategy six months ago — but that decision lives in a Slack thread, not in code</li>
                    <li>Two modules are being deprecated next quarter, and any new pipeline should route around them</li>
                </ul>
                <p>This is <strong>institutional knowledge</strong> — the accumulated wisdom of the team, scattered across people's heads, old threads, postmortems, and design documents nobody maintains. MCP can't see any of it. And without it, agents produce work that is technically correct but architecturally naive.</p>
                <p>The solution is a structured knowledge base — an LLM-maintained wiki that captures the institutional knowledge MCP can't reach. I've written about this in depth in <a href="/posts/knowledge-base-as-agentic-memory"><strong>The Knowledge Base Strategy: Giving AI Agents a Memory That Compounds</strong></a>, and I've open-sourced a template to build one: <a href="https://github.com/bmentges/knowledge-base-template" target="_blank" rel="noopener"><strong>knowledge-base-template</strong></a>.</p>
                <p>The knowledge base isn't a replacement for MCP. It's the complementary layer. MCP tells the agent what the system looks like. The knowledge base tells the agent why the system looks that way.</p>
            </section>

            
            <section class="content-section">
                <h2>The full stack: MCP + Knowledge Base + Guardrails</h2>
                <p>This is where the three pieces come together into a complete context layer for production-grade AI agent integration:</p>
                <ul>
                    <li><strong>MCP = eyes.</strong> Live infrastructure data. What the system looks like right now. Real-time, precise, infrastructure-scoped. The agent queries context servers to understand schemas, pipelines, run histories, and conventions.</li>
                    <li><strong>Knowledge Base = memory.</strong> Institutional wisdom. Why the system looks that way. Compounding, synthesized, organization-scoped. Architecture decisions, postmortem lessons, domain rules, in-flight projects. The knowledge that code alone can't convey.</li>
                    <li><strong>Guardrails = boundaries.</strong> Trust framework. What the agent can and cannot do. Three zones that earn adoption by making risk explicit. The foundation that lets the team say "yes" to agent integration instead of "maybe later."</li>
                </ul>
                <p>Each piece is valuable on its own. Together, they're the complete context infrastructure that turns generic AI tools into team-specific engineering partners.</p>

            <section class="content-section">
                <h2>Getting started</h2>
                <p>If you're reading this and thinking "this sounds like a lot," here's the pragmatic path. You don't need all three layers on day one. You need one well-chosen entry point that earns trust and demonstrates value.</p>

                <h3>Step 1: Define guardrails (day 1)</h3>
                <p>Before you build anything, write down the three zones for your team. What can agents do autonomously? What needs review? What's off-limits? This takes an hour and it's the most important hour you'll spend. Without this document, your first agent incident will kill the initiative. With it, incidents become boundary refinements.</p>

                <h3>Step 2: Build one context server (week 1)</h3>
                <p>Pick the server with the most immediate value. For most data engineering teams, that's the schema/warehouse metadata server — agents use it constantly for code generation, review, and scaffolding. Get it running, connect it to your agents via MCP, and let the team experience context-aware AI assistance. The jump from generic suggestions to infrastructure-grounded suggestions is visceral. It sells the approach faster than any slide deck.</p>

                <h3>Step 3: Build your first agent (week 2)</h3>
                <p>Start with incident context assembly. It has the highest impact-to-effort ratio and the fastest team buy-in — nobody argues with "the thing that makes 3 AM pages less painful." Keep it in the CAN DO zone (read-only context assembly, no remediation actions). Let it prove itself.</p>

                <h3>Step 4: Start the knowledge base (week 3)</h3>
                <p>Ingest your most important institutional documents — three ADRs, two postmortems, your system topology doc. Use the <a href="https://github.com/bmentges/knowledge-base-template" target="_blank" rel="noopener"><strong>knowledge-base-template</strong></a> to set up the structure. The knowledge base starts compounding from the moment you feed it the first document. Every agent you build after this point benefits from it.</p>

                <h3>Step 5: Expand (week 4+)</h3>
                <p>Add context servers. Build more agents. Ingest more institutional knowledge. The compound effect means each addition is more valuable than the last — the second agent benefits from the context the first agent's server already provides, plus whatever knowledge base content has accumulated since week 3.</p>

                <p>Measure adoption, not capability. The metric that matters isn't "how smart is the agent" — it's "how many engineers use it regularly." Smart agents that nobody trusts produce zero value. Limited agents that the team relies on daily compound into transformative tools.</p>
            </section>

            
            <section class="content-section">
                <h2>The bottom line</h2>
                <p>AI agents in engineering are production-ready. But their value is determined entirely by the context infrastructure underneath them.</p>
                <p>MCP gives agents eyes into your live infrastructure — the <em>what</em>. A knowledge base gives them memory of your organization — the <em>why</em>. Guardrails define the boundaries that earn team trust — the <em>how much</em>. Together, they're the full context layer that turns generic AI into your team's AI.</p>
                <p>The teams that will win with AI agents aren't the ones with the best models. They're the ones that build the context infrastructure to make those models genuinely understand their system. MCP is open. The <a href="https://github.com/bmentges/knowledge-base-template" target="_blank" rel="noopener">knowledge base template</a> is open. The guardrails framework is straightforward. The hard part isn't the technology — it's the organizational discipline to start with boundaries, build context systematically, and let trust compound.</p>
                <p>Start small. Start with guardrails. Build one context server. Ship one agent. Let it prove itself. Then expand.</p>
